Troubleshooting y Cheatsheet CLI

¿Qué es el CLI?

El CLI (Command Line Interface) es la pantalla negra donde escribes comandos para configurar routers y switches Cisco. No tiene botones ni menús gráficos: todo se hace escribiendo texto. Piensa en él como una conversación con el equipo: tú le das instrucciones y él las ejecuta.

Lo más importante que debes entender es que el CLI tiene diferentes "modos", como niveles en un videojuego. Cada modo te permite hacer cosas distintas, y el símbolo al final del prompt (lo que ves antes de escribir) te dice en qué modo estás.

Los 3 modos principales que SIEMPRE debes reconocer:

Modo Prompt (lo que ves) ¿Cómo llego ahí? ¿Qué puedo hacer?
User EXEC (modo usuario) Router> Es donde inicias automáticamente al conectarte Muy poco: solo comandos básicos de visualización como ping, show version. No puedes cambiar nada.
Privileged EXEC (modo privilegiado) Router# Escribes enable desde User EXEC Puedes ver TODA la configuración (show running-config), guardar cambios, reiniciar el equipo. Aún no puedes configurar.
Global Configuration (modo de configuración global) Router(config)# Escribes configure terminal desde Privileged EXEC ¡Aquí es donde configuras todo! Hostname, contraseñas, rutas, etc. Este es tu modo de trabajo principal.

¿Cómo me muevo entre modos? (Navegar por el CLI)

Imagina una escalera de 3 peldaños. Siempre empiezas en el más bajo (User EXEC) y vas subiendo:

Subir: User EXEC > → escribe enable → Privileged EXEC # → escribe configure terminal (o conf t como atajo) → Global Config (config)#

Bajar un nivel: escribe exit en cualquier modo para retroceder un paso.

Bajar DIRECTO al modo privilegiado: escribe end (o presiona Ctrl+Z) desde cualquier modo de configuración. Esto te saca de golpe al modo #.

! Ejemplo completo: navegar entre los 3 modos Router> enable Router# configure terminal Router(config)# exit Router# exit Router>
! Atajo: desde cualquier sub-modo, "end" te lleva directo al # Router(config-if)# end Router#

Los sub-modos de configuración (dentro de Global Config):

Cuando estás en Global Config (config)# y necesitas configurar algo específico (una interfaz, una VLAN, una línea de consola), entras a un sub-modo. El prompt cambia para decirte exactamente dónde estás:

Sub-modo Prompt ¿Cómo entro? ¿Para qué sirve?
Interfaz (config-if)# interface GigabitEthernet0/0 Configurar un puerto específico: IP, encender/apagar, velocidad
VLAN (config-vlan)# vlan 10 Crear o nombrar una VLAN
Router (protocolo de enrutamiento) (config-router)# router rip o router ospf 1 Configurar protocolos de enrutamiento (RIP, OSPF, EIGRP)
Línea (consola/vty) (config-line)# line console 0 o line vty 0 4 Configurar acceso por consola física o por Telnet/SSH remoto
Sub-interfaz (config-subif)# interface G0/0.10 Crear sub-interfaces para Router-on-a-Stick (inter-VLAN routing)
Truco: Usa el signo del prompt como brújula

Si alguna vez te pierdes, mira el prompt. Si ves > estás en usuario, si ves # sin paréntesis estás en privilegiado, y si ves (config-algo)# estás en algún modo de configuración. ¡El prompt SIEMPRE te dice dónde estás!

¿Por qué configuración básica?

Antes de configurar VLANs, IPs, rutas o cualquier cosa avanzada, TODO equipo Cisco (sea switch o router) necesita una configuración de seguridad básica. Es como ponerle nombre, cerradura y un letrero de advertencia a tu casa antes de empezar a decorarla por dentro.

La rúbrica de tu proyecto te quita puntos si no tienes esta configuración en CADA equipo. Así que memoriza estos comandos: los vas a repetir muchas veces.

Estos son los comandos y qué hace CADA uno:

Comando ¿Qué hace? ¿Por qué es necesario?
hostname NOMBRE Le pone un nombre al equipo. El prompt cambia de "Switch" a lo que tú pongas. Para identificar cada equipo en tu topología. Si tienes 10 switches, necesitas saber cuál es cuál.
enable secret class Crea una contraseña encriptada para entrar al modo privilegiado (#). Sin esto, cualquiera podría escribir enable y tener acceso total al equipo. La palabra "class" es la contraseña (la puedes cambiar).
banner motd & Texto & Muestra un mensaje de advertencia cuando alguien se conecta al equipo. Es un requisito legal y de la rúbrica. Advierte que el acceso no autorizado está prohibido. El símbolo & es el delimitador (puede ser cualquier carácter que no esté en tu mensaje).
line console 0 → password cisco → login Protege el acceso por cable de consola (la conexión física directa al equipo). Sin esto, alguien que conecte un cable de consola entra directo sin contraseña. El comando login activa la verificación.
line vty 0 4 → password cisco → login Protege el acceso remoto por Telnet/SSH (las 5 líneas virtuales: 0, 1, 2, 3 y 4). Sin esto, nadie podría conectarse remotamente al equipo. El rango 0 4 significa que permite hasta 5 conexiones simultáneas.
service password-encryption Encripta TODAS las contraseñas en texto plano que aparecen en la configuración. Sin esto, si alguien ve tu show running-config, puede leer las contraseñas de consola y VTY directamente. Este comando las oculta con encriptación tipo 7.

Bloque completo para un SWITCH (copia y pega):

! Paso 1: Entrar al modo privilegiado Switch> enable
! Paso 2: Entrar al modo de configuración global Switch# configure terminal
! Paso 3: Ponerle nombre al switch para identificarlo Switch(config)# hostname S1_LIMA

! Paso 5: Crear mensaje de advertencia (banner) ! El & es el delimitador, el mensaje va entre los dos & S1_LIMA(config)# banner motd & Acceso solo para personal autorizado de MIEMPRESA &
! Paso 6: Proteger el acceso por consola (cable físico) S1_LIMA(config)# line console 0 S1_LIMA(config-line)# password cisco S1_LIMA(config-line)# login S1_LIMA(config-line)# exit
! Paso 7: Proteger el acceso remoto (Telnet/SSH) ! "0 4" = 5 líneas virtuales = 5 personas pueden conectarse a la vez S1_LIMA(config)# line vty 0 4 S1_LIMA(config-line)# password cisco S1_LIMA(config-line)# login S1_LIMA(config-line)# exit

¿Por qué "enable secret" y no "enable password"?

Ambos crean una contraseña para el modo privilegiado, pero enable secret la guarda con encriptación fuerte (MD5), mientras que enable password la guarda en texto plano (o con encriptación débil tipo 7). Siempre usa enable secret — es lo que pide la rúbrica y es más seguro.

¿Para qué sirven los comandos de diagnóstico?

Cuando algo no funciona en tu red (y va a pasar muchas veces), necesitas herramientas para investigar qué está mal. Estos comandos son tus "instrumentos médicos": te permiten revisar si los cables están bien, si las IPs son correctas, si las rutas existen, y si los paquetes llegan a su destino.

Regla de oro: Después de configurar algo, SIEMPRE verifica con estos comandos que funcionó. No asumas que está bien solo porque no te dio error.

Comando ¿Qué muestra? ¿Cuándo lo uso?
ping [ip-destino] Envía 5 paquetes ICMP al destino y muestra si llegan. ! = éxito, . = fallo. Ejemplo: !!!!! = 5/5, todo perfecto. Siempre. Es tu primer comando de diagnóstico. ¿Configuraste una IP? Haz ping. ¿Creaste una ruta? Haz ping. ¿Algo no funciona? Haz ping.
tracert [ip] (desde PC) o traceroute [ip] (desde router) Muestra CADA salto (router) que recorre el paquete hasta llegar al destino, con la IP de cada salto. Cuando el ping falla y quieres saber DÓNDE se rompe la comunicación. Te dice exactamente en qué router se detiene el paquete.
show ip interface brief Una tabla resumen de TODAS las interfaces del equipo: nombre, IP asignada, y si está UP (activa) o DOWN (apagada). Para verificar que las IPs están bien asignadas y que las interfaces están encendidas. Si ves "administratively down", significa que la interfaz está apagada con shutdown.
show ip route La tabla de enrutamiento completa del equipo. Muestra todas las redes que el equipo sabe cómo alcanzar. C = conectada directamente, S = ruta estática, R = aprendida por RIP, O = OSPF. Cuando un ping falla entre redes diferentes. Si la red destino no aparece en esta tabla, ¡el router no sabe cómo llegar! Necesitas agregar una ruta.
show vlan brief Lista todas las VLANs creadas en el switch y qué puertos pertenecen a cada una. Después de crear VLANs y asignar puertos. Verifica que cada puerto esté en la VLAN correcta. Si un puerto no aparece en ninguna VLAN de tu tabla, probablemente está en la VLAN 1 (por defecto).
show interfaces trunk Muestra qué puertos están configurados como trunk y qué VLANs están permitidas en cada trunk. Cuando las PCs de una VLAN no se comunican entre switches. Si el trunk no existe o no permite tu VLAN, el tráfico no pasa.
show running-config Muestra TODA la configuración actual del equipo (la que está en RAM). Es el "archivo completo" de todo lo que has configurado. Cuando quieres revisar todo de una vez, o cuando buscas un error y no sabes dónde está. Puedes usar show running-config | section interface para filtrar secciones.
show interfaces [interfaz] Información detallada de una interfaz específica: velocidad, duplex, errores, paquetes enviados/recibidos. Cuando necesitas datos técnicos específicos de un puerto, como verificar si tiene errores o cuántos paquetes ha procesado.

El truco del "do" — ¡tu mejor amigo!

Los comandos show y ping normalmente solo funcionan en modo privilegiado (#). Pero cuando estás configurando en modo (config)#, no quieres salir de ahí solo para verificar algo.

La solución: agrega la palabra do antes del comando. Esto le dice al equipo "ejecuta este comando como si estuviera en modo privilegiado, aunque estoy en config".

! Sin "do": te da error porque estás en modo config S1_LIMA(config)# show ip interface brief ^ % Invalid input detected at '^' marker.
! Con "do": ¡funciona perfecto sin salir de config! S1_LIMA(config)# do show ip interface brief
! Más ejemplos del truco "do" S1_LIMA(config)# do show vlan brief S1_LIMA(config)# do ping 10.192.8.1 S1_LIMA(config)# do show running-config S1_LIMA(config-if)# do write
Truco extra: el primer ping a veces falla

No te asustes si ves .!!!! (el primer paquete falló, los otros 4 funcionaron). Esto es normal — el primer paquete se pierde porque el equipo está resolviendo la dirección MAC con ARP. Haz ping de nuevo y debería salir !!!!! (5 de 5).

⚠️ SI NO GUARDAS, PIERDES TODO

Cuando cierras Packet Tracer o reinicias un equipo, toda la configuración que no hayas guardado DESAPARECE. Hemos visto compañeros perder horas de trabajo por no guardar. No seas uno de ellos. Guarda después de CADA cambio importante.

Entendiendo RAM vs NVRAM (¿Dónde vive tu configuración?)

Tu equipo Cisco tiene dos tipos de memoria importantes:

RAM (Running-Config): Es la memoria temporal, como la memoria RAM de tu computadora. Aquí está la configuración que estás usando en este momento. Todo lo que configuras va aquí primero. Pero si apagas el equipo, se borra.

NVRAM (Startup-Config): Es la memoria permanente, como el disco duro. Lo que guardes aquí sobrevive cuando apagas y prendes el equipo. Cuando el equipo enciende, copia el Startup-Config a la RAM.

Analogía: Imagina que estás escribiendo un documento en Word. Lo que ves en pantalla es la RAM (running-config). Si se va la luz, pierdes todo. Pero si le das "Guardar" (Ctrl+S), lo grabas en el disco duro (NVRAM/startup-config). La próxima vez que abras el archivo, está ahí.

Los 3 comandos para guardar (todos hacen lo mismo):

Comando ¿Desde dónde lo ejecuto? Nota
copy running-config startup-config Modo privilegiado # El comando "oficial" y completo. Te preguntará el nombre del archivo destino — solo presiona Enter para aceptar el nombre por defecto.
write memory (o solo write / wr) Modo privilegiado # Atajo más corto. Hace exactamente lo mismo que el anterior. Muchos profesores prefieren este porque es más rápido de escribir.
do wr Desde CUALQUIER modo de configuración (config)# ¡El más práctico! Puedes guardar sin salir del modo config. Usa el truco "do" que aprendimos arriba.
! Opción 1: Comando oficial (desde modo privilegiado) S1_LIMA# copy running-config startup-config Destination filename [startup-config]? ← solo presiona Enter Building configuration... [OK]
! Opción 2: Atajo rápido (desde modo privilegiado) S1_LIMA# wr Building configuration... [OK]
! Opción 3: Guardar sin salir de config (¡EL MEJOR!) S1_LIMA(config)# do wr Building configuration... [OK]
Consejo: Guarda también el archivo .pkt de Packet Tracer

Guardar la configuración dentro del equipo (running → startup) protege contra reinicios del equipo virtual. Pero también debes guardar el archivo .pkt de Packet Tracer (Ctrl+S en la ventana de Packet Tracer) para no perder tu topología completa si la aplicación se cierra.

Diseño Lógico IP y VLSM (La Base de Todo)

La dirección IP — Tu "dirección de casa" en la red

Así como tu casa tiene una dirección (Av. Javier Prado 1234, Lima), cada dispositivo en una red necesita una dirección IP para que los demás sepan dónde encontrarlo. Sin una IP, un dispositivo es "invisible" en la red.

Una dirección IP (versión 4, que es la que usamos) se ve así: 10.192.8.1. Son 4 números separados por puntos, donde cada número va de 0 a 255. En total son 32 bits (4 grupos de 8 bits cada uno).

La máscara de subred — ¿Qué parte es el "barrio" y qué parte es la "casa"?

Cada dirección IP tiene dos partes, pero no puedes verlas a simple vista. Necesitas la máscara de subred para separarlas:

Parte de Red (Network): Identifica el "barrio" o red a la que perteneces. Todos los dispositivos en la misma red comparten esta parte. Es como el nombre de tu calle.

Parte de Host: Identifica tu dispositivo específico dentro de esa red. Es como el número de tu casa.

Ejemplo con analogía: Si la dirección es "Calle Los Olivos, Casa #15", entonces "Calle Los Olivos" es la parte de red y "#15" es la parte de host. La máscara te dice dónde termina el nombre de la calle y empieza el número de casa.

La máscara se puede escribir de dos formas: como decimal (255.255.255.0) o como prefijo (/24). El /24 significa que los primeros 24 bits son la parte de red.

Tabla de referencia de máscaras — Esta es la tabla más importante que usarás en todo el proyecto. Memoriza al menos /24, /25, /26 y /30:

Prefijo Máscara Decimal Hosts Totales Hosts Utilizables ¿Para qué se usa típicamente?
/24 255.255.255.0 256 254 Departamentos grandes (ventas con muchos empleados)
/25 255.255.255.128 128 126 Departamentos medianos
/26 255.255.255.192 64 62 Departamentos pequeños
/27 255.255.255.224 32 30 VLAN de gestión u oficinas pequeñas
/28 255.255.255.240 16 14 Grupos muy reducidos, servidores
/29 255.255.255.248 8 6 Segmentos mínimos con pocos equipos
/30 255.255.255.252 4 2 Enlaces punto a punto entre routers (WAN)

¿Por qué "Hosts Utilizables" son menos que "Hosts Totales"?

Porque en cada subred, 2 direcciones están reservadas y no se pueden asignar a dispositivos:

La primera IP de la subred es la dirección de red (identifica a toda la subred, como el nombre del barrio). Ejemplo: 10.192.8.0

La última IP de la subred es la dirección de broadcast (envía mensajes a TODOS los dispositivos de la subred a la vez). Ejemplo: 10.192.8.255

Por eso la fórmula es: Hosts utilizables = 2n - 2, donde n es el número de bits para hosts (32 - prefijo). Para /24: 28 - 2 = 256 - 2 = 254 hosts.

¿Cuál es la primera IP utilizable? → ¡Esa es tu Gateway!

Por convención, la primera IP utilizable de cada subred (la dirección de red + 1) se asigna al gateway (la "puerta de salida" de esa red, normalmente un router o switch L3). Ejemplo: si la red es 10.192.8.0/24, el gateway es 10.192.8.1.

FLSM vs VLSM — La diferencia explicada fácil

FLSM (Fixed Length Subnet Masking): Imagina que tienes un pastel y lo cortas en pedazos iguales. No importa si un invitado tiene mucha hambre y otro poca — todos reciben el mismo tamaño. Funcional, pero desperdicias mucho pastel si los pedazos son demasiado grandes para algunos.

VLSM (Variable Length Subnet Masking): Ahora imagina que cortas el pastel en pedazos de diferentes tamaños según el hambre de cada invitado. El hambriento recibe un pedazo grande, y el que no tiene mucha hambre recibe uno chico. No desperdicias nada.

En redes: FLSM usa la misma máscara para todas las subredes (por ejemplo, todas /24). VLSM usa máscaras diferentes según cuántos hosts necesita cada departamento (/24 para uno grande, /26 para uno pequeño, /30 para un enlace WAN).

¿Por qué usamos VLSM en el proyecto? Porque la rúbrica lo pide, porque es lo que se usa en el mundo real, y porque con FLSM desperdiciaríamos miles de direcciones IP.

El proceso paso a paso para hacer VLSM

Sigue estos pasos en orden siempre. Si cambias el orden, el subnetting puede traslarse y causar solapamiento de redes:

Paso 1: Calcula el crecimiento proyectado. Multiplica el número actual de hosts por el factor de crecimiento:

• Sede principal (Lima): Hosts actuales × 1.25 (25% de crecimiento a 10 años)

• Sucursales: Hosts actuales × 1.20 (20% de crecimiento a 10 años)

Paso 2: Ordena TODAS las subredes de MAYOR a MENOR número de hosts (después de aplicar el crecimiento). La subred que necesite más hosts va PRIMERO.

Paso 3: Para cada subred, encuentra la máscara más pequeña que cubra la cantidad de hosts necesarios. Usa la tabla de prefijos que vimos arriba.

Paso 4: Asigna las direcciones en orden, una después de otra, sin dejar huecos y sin traslapes. La siguiente subred empieza donde termina la anterior.

¡SIEMPRE redondea hacia ARRIBA!

Si un departamento tiene 50 empleados y con el crecimiento del 25% necesitas 50 × 1.25 = 62.5 hosts, redondeas a 63. Luego buscas la máscara que cubra al menos 63 hosts utilizables → eso es un /26 (que da 62 hosts utilizables). ¡Pero 62 < 63! Entonces necesitas un /25 (126 hosts). Siempre verifica que los hosts utilizables ≥ hosts necesarios.

Ejemplo rápido de cálculo de crecimiento:

Departamento Hosts Actuales Factor Cálculo Hosts Necesarios (redondeado ↑) Máscara necesaria
Ventas (Lima) 200 × 1.25 200 × 1.25 = 250 250 /24 (254 utilizables ≥ 250 ✓)
Admin (Lima) 100 × 1.25 100 × 1.25 = 125 125 /25 (126 utilizables ≥ 125 ✓)
Finanzas (Lima) 50 × 1.25 50 × 1.25 = 62.5 → 63 63 /26 (62 utilizables) ¡NO alcanza! → /25 o ajustar a 50 exactos con /26
El orden importa: ¡De mayor a menor!

Si no ordenas de mayor a menor, podrías asignar una subred pequeña en medio y luego no tener espacio contiguo para una subred grande. Es como tratar de estacionar un camión en un espacio que dejaste entre dos autos: no cabe. Siempre estaciona primero el camión (subred grande), luego los autos (subredes medianas), y al final las motos (subredes chicas).

¿Cómo leer esta tabla? (Ejemplo con Ventas)

Tomemos la fila de Ventas para explicar cada columna:

Dirección de Red (10.192.8.0): Es la primera dirección de esta subred. NO se asigna a ningún dispositivo. Es solo el "nombre" de la red.

Gateway (10.192.8.1): Es la primera IP utilizable, y la asignamos al router o switch L3 que conecta esta VLAN con el resto de la red. Las PCs de Ventas configuran esta IP como su "puerta de enlace predeterminada" para poder comunicarse con otras redes.

Última IP útil (10.192.8.254): Es la última dirección que puedes asignar a un dispositivo en esta subred.

Broadcast (10.192.8.255): Es la última dirección de la subred. Se usa para enviar mensajes a TODOS los dispositivos de la red. NO se asigna a ningún dispositivo.

Máscara (255.255.255.0 = /24): Le dice a cada dispositivo qué parte de la IP es "red" y qué parte es "host". La configuras en cada PC y cada interfaz de router/switch.

Las otras sedes siguen la misma lógica

La Libertad, Ica, Huánuco y Puno tienen la misma estructura de VLANs (10, 20, 30, 99, 999), pero con diferentes bloques de direcciones IP dentro del rango 10.192.0.0/16. El proceso de subnetting es idéntico — solo cambian los números. Si entiendes Lima, entiendes todas.

VLAN de Gestión (VLAN 99) — ¿Para qué sirve?

Las VLANs 10, 20 y 30 son para las computadoras de los empleados (Ventas, Administración, Finanzas). Pero los equipos de red (switches, routers, access points) también necesitan una IP para que los puedas administrar remotamente.

La VLAN de Gestión (VLAN 99) es una VLAN especial dedicada exclusivamente a los equipos de red. No se conectan computadoras de usuarios aquí — solo los switches, routers y access points reciben una IP de esta VLAN.

¿Por qué separarla? Por seguridad. Si un empleado malintencionado intenta acceder a los switches desde su PC, no puede porque está en otra VLAN. El tráfico de gestión viaja aislado del tráfico de usuarios.

¿Cuántas IPs necesita la VLAN de Gestión?

Necesitas 1 IP por cada equipo de red que quieras administrar. Cuenta todos los dispositivos de red en tu sede:

Fórmula: Total de IPs = Routers + Switches L3 + Switches L2 + Access Points + (margen de crecimiento)

Ejemplo práctico para la sede Lima:

Tipo de equipo Cantidad ¿Necesita IP de gestión?
Router principal 1 Sí (para administrarlo remotamente)
Switch L3 (Core/Distribución) 1 Sí (es el gateway de las VLANs)
Switches L2 (Acceso) 3 Sí (uno por piso o departamento)
Access Points (Wi-Fi) 2 Sí (para configurar la red inalámbrica)
Total 7 Con crecimiento (× 1.25) = 8.75 → 9 hosts

Con 9 hosts necesarios, un /28 (14 hosts útiles) sobra, pero en nuestro diseño usamos /27 (30 hosts útiles) para dejar más margen de crecimiento. Es mejor que sobre espacio y no que falte.

Enlaces /30 — Conexiones punto a punto entre routers

Cuando conectas dos routers entre sí (por ejemplo, el router de Lima con el router de La Libertad), ese cable solo conecta 2 dispositivos: un extremo del router A y un extremo del router B. No necesitas 254 IPs para eso, ¡solo necesitas 2!

Para eso existe la máscara /30 (255.255.255.252), que da exactamente 2 hosts utilizables. Es la máscara más eficiente posible para un enlace punto a punto: una IP para cada lado del cable.

Anatomía de un bloque /30:

• 1ra dirección: Dirección de red (no se usa)

• 2da dirección: IP del router A (primer host utilizable)

• 3ra dirección: IP del router B (segundo host utilizable)

• 4ta dirección: Broadcast (no se usa)

Usamos el rango 172.16.0.0/30 para los enlaces punto a punto. Cada enlace WAN consume un bloque /30 de 4 direcciones:

Lima ↔ Ica 172.16.0.4/30 172.16.0.5 (R_LIMA) 172.16.0.6 (R_ICA) 172.16.0.7
Lima ↔ Huánuco 172.16.0.8/30 172.16.0.9 (R_LIMA) 172.16.0.10 (R_HUANUCO) 172.16.0.11
Lima ↔ Puno 172.16.0.12/30 172.16.0.13 (R_LIMA) 172.16.0.14 (R_PUNO) 172.16.0.15

¿Ves el patrón?

Cada bloque /30 usa exactamente 4 direcciones. El siguiente bloque empieza donde terminó el anterior:

• Primer enlace: .0, .1, .2, .3 → siguiente empieza en .4

• Segundo enlace: .4, .5, .6, .7 → siguiente empieza en .8

• Tercer enlace: .8, .9, .10, .11 → siguiente empieza en .12

Siempre sumas 4 al número de red anterior para obtener el siguiente bloque /30. Fácil de memorizar.

No confundas las redes de tránsito con las redes de las sedes

Las IPs 10.192.x.x son para los dispositivos dentro de las sedes (PCs, impresoras, switches de acceso). Las IPs 172.16.0.x son para los cables entre routers (enlaces WAN). Son dos mundos separados. Un router tiene una "pata" en cada mundo: interfaces LAN con IPs 10.192.x.x e interfaces WAN con IPs 172.16.0.x.

También se usan /30 dentro de cada sede

El enlace entre el router de la sede y el switch L3 Core también usa un bloque /30, porque es igualmente un enlace punto a punto entre solo 2 equipos. No tiene sentido "desperdiciar" un /24 (254 IPs) para un cable que solo conecta 2 dispositivos.
