"La Regla de la X" - ¡Importante!

La dirección de red base de la multinacional es 10.19X.0.0/16. La "X" debe ser reemplazada obligatoriamente por el número de tu grupo. ¡Si olvidas esto, habrá duplicidad de IPs y tu red colapsará al integrarse al final del semestre!

01

Diseño Lógico IP y VLSM

Fundamento Teórico

Las redes "Classful" usan máscaras fijas, desperdiciando miles de IPs. VLSM (Variable Length Subnet Mask) permite "hacer subredes de una subred", asignando máscaras variables según la necesidad exacta de cada departamento, maximizando la eficiencia.

Según la rúbrica, no puedes dimensionar la red con los hosts actuales. Debes aplicar un aumento proporcional del 25% a cada VLAN.

Ejemplo Matemático Exacto (Sede Lima):

• Administración: Tienes 100 hosts actuales. Le calculas el aumento del 25% (`100 * 1.25`) y tienes que dimensionar para 125 hosts. Necesitarás una subred /24 (254 útiles).

• Logística: Tienes 25 hosts actuales. Con el aumento serían `31.25`. Como no puede ser decimal, siempre se redondea hacia arriba, por lo que te quedarías con 32 hosts. Necesitarás una subred /26 (62 útiles).

* Aplica este mismo cálculo y redondeo hacia arriba para absolutamente todos los demás departamentos.

No desperdicies IPs en Enlaces Seriales

Para los enlaces WAN (Punto a Punto entre sedes), debes usar SIEMPRE subredes /30. Una máscara /30 solo provee 2 direcciones útiles, exactamente las que necesitas para conectar dos routers. Si usas /24 o FLSM, te descontarán puntos.

El proyecto utiliza un Switch Multicapa (Ej: Catalyst 3650). Este dispositivo de Capa 3 asume el rol de Gateway para todas las VLANs, creando "Interfaces Virtuales (SVI)".

04

Enlaces WAN y Autenticación PPP CHAP

Fundamento Teórico

En enlaces seriales WAN, el protocolo por defecto de Cisco es HDLC. Sin embargo, para mayor compatibilidad y seguridad cambiamos el encapsulamiento a PPP (Point-to-Point Protocol).

El laboratorio nos enseña dos métodos de autenticación: PAP envía la contraseña en texto plano (inseguro), mientras que CHAP (Challenge Handshake Authentication Protocol) usa un desafío encriptado de 3 vías (mucho más seguro). Es por eso que el proyecto exige CHAP.

Regla de Nombres de Usuario

Si configuras el ROUTER A que conecta con el ROUTER B, el comando debe ser `username ROUTER_B secret miclave`. La clave (`secret`) debe ser EXACTAMENTE la misma en ambos routers.

! CONFIGURACIÓN EN ROUTER LIMA (Conectando hacia Ica): Router_Lima(config)# username Router_Ica secret ciscoclass Router_Lima(config)# interface s0/0/0 Router_Lima(config-if)# ip address 10.81.36.1 255.255.255.252 Router_Lima(config-if)# encapsulation ppp Router_Lima(config-if)# ppp authentication chap Router_Lima(config-if)# no shutdown
! CONFIGURACIÓN EN ROUTER ICA (Conectando hacia Lima): Router_Ica(config)# username Router_Lima secret ciscoclass Router_Ica(config)# interface s0/0/0 Router_Ica(config-if)# ip address 10.81.36.2 255.255.255.252 Router_Ica(config-if)# encapsulation ppp Router_Ica(config-if)# ppp authentication chap Router_Ica(config-if)# no shutdown

Enrutamiento RIPv2 y Rutas Estáticas

RIPv2 es un protocolo de vector distancia que compartirá automáticamente las rutas entre nuestras sedes. Las Rutas Estáticas Flotantes nos permiten tener un ISP de respaldo que solo se activará si el ISP principal falla (se logra dándole una distancia administrativa mayor a 1).

! 1. CONFIGURAR RIPv2 EN TODOS LOS ROUTERS Y CORES: Router(config)# router rip Router(config-router)# version 2 ! Debes declarar las redes directamente conectadas. Por ejemplo, si el router tiene la 10.81.36.0: Router(config-router)# network 10.81.36.0 Router(config-router)# network 10.81.0.0 ! CRÍTICO: Prevenir que RIP destruya el VLSM sumarizando las redes Router(config-router)# no auto-summary Router(config-router)# exit
! 2. RUTAS ESTÁTICAS AL INTERNET (Solo en el Router Frontera / Borde): ! Ruta Predeterminada Principal por el ISP 1 Router(config)# ip route 0.0.0.0 0.0.0.0 209.165.200.226 ! Ruta Flotante de Respaldo por el ISP 2 (Distancia Administrativa 5) Router(config)# ip route 0.0.0.0 0.0.0.0 209.165.201.2 5

Servicios de Red (La Magia de Capa 7)

DNS (Domain Name System)

Las computadoras se comunican usando IPs numéricas, pero los humanos usamos nombres (ej. `www.google.com`). El servidor DNS actúa como una gigantesca guía telefónica que traduce estos nombres a IPs.

WEB (HTTP/HTTPS)

El Servidor WEB procesa aplicaciones del lado del servidor. Generalmente usa el protocolo HTTP (puerto TCP 80) o HTTPS (puerto TCP 443) para enviar el código de la página al navegador del cliente.

CORREO (SMTP, POP3, IMAP)

Trabaja como una oficina postal. El protocolo SMTP (Puerto TCP 25) se encarga de transportar o enviar el mensaje (MTA). El protocolo POP3 (Puerto TCP 110) se encarga de descargar y recuperar el correo desde el servidor hacia la PC cliente. IMAP es similar pero mantiene la copia sincronizada en el servidor.

FTP y TCP

FTP permite la transferencia de archivos. Utiliza el protocolo TCP, el cual orienta la conexión y garantiza la fiabilidad del envío de los paquetes (a diferencia de UDP que es más rápido pero no garantiza la entrega).

Seguridad (Las 5 Políticas ACL y SSH)

Listas de Control de Acceso (ACL)

Una ACL actúa como un cortafuegos interno. Filtra el tráfico permitiéndolo (`permit`) o denegándolo (`deny`).

• ACL Estándar: Solo filtran en base a la IP de origen (Números del 1 al 99).

• ACL Extendida: Filtran en base a origen, destino y protocolo o puerto específico (Números del 100 al 199).

La Máscara Wildcard (Comodín)

Al escribir ACLs no usamos máscaras de subred normales, usamos Wildcards. Si lo traducimos a binario:

• El 0 significa "ME IMPORTA" (Tiene que coincidir el número exacto).

• El 1 significa "NO ME IMPORTA" (Puede ser cualquier valor).

Ejemplo del Laboratorio 12:
IP: 192.168.10.0 | Wildcard: 0.0.0.255

Explicación: "De la IP me importan los tres primeros octetos (192.168.10) porque tienen 0. El último octeto tiene 255 (puros 1 en binario), así que NO ME IMPORTA". Resultado: Abarca desde la IP .1 hasta la .254 de esa red.

Aplica esta ACL Extendida en el CORE para cumplir la rúbrica al 100%. (Reemplaza las IPs con las de tus servidores).

Bloqueo Implícito (Deny Any Any)

Toda ACL tiene un bloqueo automático al final. Si omites el comando permit ip any any como última regla de tu lista, bloquearás todo el tráfico (pings, etc).

! Entramos a configurar la ACL Extendida CORE_LIMA(config)# ip access-list extended POLITICAS_LIMA ! POLÍTICA 1 - FTP: Permitir FTP (21) solo hacia el Servidor FTP (ej. 10.81.43.38) CORE_LIMA(config-ext-nacl)# permit tcp any host 10.81.43.38 eq 21 ! POLÍTICA 2 - WEB: Permitir solo HTTP (80) y HTTPS (443) CORE_LIMA(config-ext-nacl)# permit tcp any any eq 80 CORE_LIMA(config-ext-nacl)# permit tcp any any eq 443 ! POLÍTICA 3 - DHCP: Permitir UDP puertos 67 y 68 CORE_LIMA(config-ext-nacl)# permit udp any any eq 67 CORE_LIMA(config-ext-nacl)# permit udp any any eq 68 ! POLÍTICA 4 - CORREO: Permitir SMTP (25) y POP3 (110) hacia el Server Mail (ej. .37) CORE_LIMA(config-ext-nacl)# permit tcp any host 10.81.43.37 eq 25 CORE_LIMA(config-ext-nacl)# permit tcp any host 10.81.43.37 eq 110 ! CRÍTICO: Evitar que el Deny Implícito bloquee todo lo demás (pings) CORE_LIMA(config-ext-nacl)# permit ip any any CORE_LIMA(config-ext-nacl)# exit ! POLÍTICA 5 - SSH GESTIÓN: Crear la clave RSA y el usuario CORE_LIMA(config)# ip domain-name lima.com CORE_LIMA(config)# crypto key generate rsa (Digita 1024) CORE_LIMA(config)# username adminLima privilege 15 secret adminLima CORE_LIMA(config)# ip ssh version 2 ! Crear ACL Estándar para asegurar que solo la PC ADMIN acceda CORE_LIMA(config)# ip access-list standard ACL_GESTION CORE_LIMA(config-std-nacl)# permit host 10.81.42.193 (IP de tu PC Admin) CORE_LIMA(config-std-nacl)# exit ! Aplicar la restricción SSH a las líneas virtuales CORE_LIMA(config)# line vty 0 4 CORE_LIMA(config-line)# transport input ssh CORE_LIMA(config-line)# login local CORE_LIMA(config-line)# access-class ACL_GESTION in
